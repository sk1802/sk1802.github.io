Adding Batch DMA Support to the Linux Kernel

What the problem is, how I tried to solve it, what the reviewers taught me, and where it goes next.

Sumit Kumar Upstream Linux Kernel 2026
01 — What

The Problem

The Linux DMA engine exposes dmaengine_prep_slave_sg() for scatter-gather transfers. It works well when you have a single source/destination relationship — one base address, multiple buffers hanging off it.

But there is a real gap: what if you need to move N independent buffers, each with its own source and destination address, in a single DMA transaction?

Today's answer: submit N separate transactions. N interrupts. N submission overheads. For high-throughput workloads this is measurable CPU tax at the kernel level.

02 — Why

Motivation & The Math

A Synopsys DesignWare eDMA controller supports linked-list (LL) mode — a hardware mechanism to chain multiple transfer descriptors into a single DMA transaction. The controller walks the linked list autonomously and fires one completion interrupt at the end, regardless of how many descriptors were chained.

Each LL page is typically 4KB. The number of burst descriptors per page is determined by the DMA burst size configured at hardware initialization. This sets the maximum batch size:

Interrupt Overhead — Before vs After
Before: N transfers → N completion interrupts → N kernel wakeups
After:   N transfers → 1 LL transaction → 1 completion interrupt
Savings = N − 1  |  64 buffers → 63 interrupts eliminated  |  256 buffers → 255 gone
Max batch = floor(LL page size / descriptor size) × number of LL pages

The savings scale linearly with N. For workloads that batch tens or hundreds of buffers, this is the difference between a constant interrupt rate and one that explodes with throughput.

03 — How (Initial Approach)

First Design

My initial approach extended struct scatterlist with an optional destination DMA address field, gated behind a new Kconfig option CONFIG_NEED_SG_DMA_DST_ADDR: 

+#ifdef CONFIG_NEED_SG_DMA_DST_ADDR
+	dma_addr_t	dma_dst_address;
+#endif

Each SG entry would then carry both a source (sg_dma_address) and a destination (sg_dma_dst_address). A new API — dmaengine_prep_batch_sg_dma() — would accept this extended list and program the eDMA linked-list hardware in a single transaction.

The DesignWare eDMA driver was extended with a new transfer type EDMA_XFER_DUAL_ADDR_SG, and the MHI endpoint driver gained a read_batch() callback with both a DMA-optimized path and a CPU-copy fallback via IATU. 

static struct dma_async_tx_descriptor *
dw_edma_device_prep_batch_sg_dma(struct dma_chan *dchan,
                                 struct scatterlist *sg,
                                 unsigned int nents,

                                 unsigned long flags)
04 — Update (Reviewer Feedback)

What the Maintainers Said

The upstream review was direct and, on reflection, correct. Two key points landed:

Robin Hood — DMA maintainer
"Eww, no, what does this even mean? Is the regular dma_addr somehow implicitly a 'source' now? How could the single piece of memory represented by page_link/offset/length have two different DMA addresses? How are both the DMA mapping code and users supposed to know which one is relevant in any particular situation?"
The issue: Extending struct scatterlist with a second DMA address creates a contract ambiguity throughout the DMA mapping layer. Drivers and the mapping code have no way to know which address field is relevant in which context. It's a layering violation.
Vinod Koul — DMA engine maintainer
"Looks like you want to bring back dmaengine_prep_dma_sg() — see commit c678fa66341c. If this is memcpy why are you talking about dma_dst_address which is a slave field?"
The issue: This exact problem was solved before. Commit c678fa66341c introduced dmaengine_prep_dma_sg() and was later reverted. The right path is to revive that approach — not reinvent it with a different shape and a new layering problem.

Both reviewers were right. I was not aware of that commit's history when I sent the patch.

05 — Final Design

Revised Approach

The revised direction: revive dmaengine_prep_dma_sg() from commit c678fa66341c, integrate it properly with the current codebase, and implement it in the DesignWare eDMA driver.

This avoids the struct scatterlist layering problem entirely. The API surface becomes cleaner — no implicit dual-address fields, no hidden Kconfig gates for the mapping layer to worry about. The implementation in dw-edma carries the dual-address logic internally where it belongs.

Revised series is in progress. The eDMA and MHI endpoint pieces remain structurally the same — only the API contract at the dmaengine layer changes.

06 — Current Upstream Use Case

MHI Ring Caching & Wraparound

The immediate upstream justification lives in the MHI (Modem Host Interface) endpoint stack. MHI uses circular ring buffers in host memory. A ring wraparound — where valid descriptor data spans two non-contiguous memory regions — is a routine occurrence.

Previously, reading a wrapped ring required two separate read_sync() calls: two DMA transactions, two interrupts, two submission overheads, just to read one logical ring's data.

With the new read_batch() callback, both regions are submitted as a single batched DMA transaction — one interrupt, one completion. The MHI endpoint driver implements two variants:

pci_epf_mhi_edma_read_batch()

DMA-optimized path. Builds a scatterlist where each entry maps one buffer region — source address from host memory, destination address in device memory — and submits the entire batch via the new DMA API. Completion is handled by a single callback.

pci_epf_mhi_iatu_read_batch()

CPU-copy fallback for non-DMA configurations. Sequentially calls pci_epf_mhi_iatu_read() for each buffer in the array. Same interface, degraded path.

The ring caching logic in the MHI stack now calls read_batch() uniformly for all ring reads — the wraparound case is no longer special-cased with extra logic.

07 — Future Use Cases

What This Unlocks Generally

The MHI ring case is the upstream anchor, but the pattern is general. Any workload moving N independent memory regions benefits — interrupt savings scale with N in all cases.

Model Load / Unload

Loading a large model means transferring many independent weight tensors. Each has its own source and destination. Today that's N transactions. With batch DMA it's one — regardless of N.

Inference Context Switching

Multi-tenant inference systems swap models on every request. Batching the unload and reload transfers cuts interrupt pressure on every switch, at the frequency of inference itself.

Ring-Based Protocol Stacks

Any protocol using circular rings — PCIe endpoint stacks, network device rings, NVMe queues — hits the wraparound problem. Batch DMA makes it a single transaction everywhere.

Many-to-Many Memory Transfers

The existing prep_slave_sg() assumes a fixed destination. Removing that constraint enables scatter-gather with fully independent destinations — a new primitive the kernel currently lacks.

For workloads batching 64 or 256 buffers at a time, the gap between N interrupts and 1 is measurable latency and CPU overhead at the kernel level. The interrupt handler itself is cheap — but scheduling, context switches, and cache pressure from N wakeups are not.

08 — Learnings

What This Process Taught Me

  • 01 Know the subsystem history before touching core data structures. Extending struct scatterlist affects every driver and every architecture that uses DMA mapping. Commits like c678fa66341c exist because this exact territory was navigated before. Read the git log, read the revert message, understand why it was undone.
  • 02 Layering violations surface quickly in upstream review. Adding a slave-mode field to a generic DMA structure created an ambiguity the mapping layer couldn't resolve. The right fix was not to patch around it but to use the correct abstraction layer.
  • 03 A concrete upstream user makes all the difference. The original dmaengine_prep_dma_sg() was reverted partly due to lack of real users. The MHI ring wraparound case provides a concrete, in-tree user that justifies the API's existence.
  • 04 Upstream review is the process working correctly. Maintainer feedback caught a real design issue before it landed in mainline. The right response is to understand why, fix it, and resubmit — not to argue the point.
  • 05 The performance case is hardware-constrained, not tunable. The 170x-class interrupt reduction is not a software optimization — it's a consequence of DMA burst size and LL page size fixed at hardware configuration. Understanding those constraints is what makes the optimization both correct and bounded.
